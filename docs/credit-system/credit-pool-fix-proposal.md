# Credit Pool 扣费修复方案（草案）

## 1. 范围与目标

- 确保积分扣费仅发生在余额为正的积分池，杜绝负债池继续扣费。
- 保障扣费与退款能够通过返回的订单信息一一对应，避免再生成孤立积分池。
- 在不处理既有历史脏数据的前提下，保证后续所有新流水的正确性与可追溯性。

## 2. 现状问题概述

1. `decreaseCredits` 函数把用户所有积分流水混合累加，导致一旦累加值足够就会在当前记录上扣费，即使该池已负余额。
2. 图片生成等业务线从扣费函数拿不到订单号和过期时间，退款时被迫创建新的积分池，无法回溯原订单。
3. 并发下缺乏事务保护，存在被其他请求抢占余额的风险，进一步加剧负债池问题。

## 3. 方案概述

### 3.1 积分扣费逻辑调整

- 新增 `groupCreditsByPool(user_uuid)` 数据访问层逻辑：
  - 以 `(order_no, expired_at)` 为维度聚合积分流水，过滤掉过期记录。
  - 聚合后只返回 `balance > 0` 的积分池，结果按 `expired_at NULLS FIRST` 升序排序，实现先到期先消耗（FIFO）。
  - 聚合查询建议使用数据库侧的 CTE 或视图，在 `WHERE user_uuid = $1` 条件下利用复合索引顺序扫描，避免全表读。
- 扣费流程：
  1. 在事务中先执行聚合查询，并带上 `credits_needed` 进行窗口累计求和，仅挑选能覆盖本次扣减所需的最小池集合。
  2. 对挑选出的池执行 `SELECT ... FOR UPDATE`，锁定对应 `(user_uuid, order_no, expired_at)` 的记录集合，而不是锁全量积分池。
  3. 构造 `deductions` 列表，并在同一事务中一次性写入扣费流水。
- 扣费函数返回本次消耗涉及的池明细 `[{ order_no, expired_at, deducted }]`，**所有池信息都必须用于退款**，按比例退回原池，避免积分分配错误。

### 3.2 接口返回值标准化

- `decreaseCredits` 改为显式返回：
  ```ts
  type DeductResult = {
    pools: Array<{
      order_no: string;
      expired_at?: string;
      deducted: number;
    }>;
    totalDeducted: number;
  };
  ```
- 调用方必须消费 `pools` 数组中的**所有池信息**，遍历每个池进行退款，确保退款金额准确退回原扣费池。

### 3.3 退款流程调整

- 图片生成、视频生成等 API：
  - 扣费后立即把 `DeductResult` 完整写入任务 `metadata.credit_deduction`，包含 `pools` 数组和 `totalDeducted`。
  - 退款时若缺少 metadata，直接抛错并记录日志，让上游重试或人工补录，避免新建无关联积分池。
  - 退款时**必须遍历 `credit_deduction.pools` 数组**，对每个池调用 `increaseCredits`，退款金额为 `pool.deducted`，确保原池余额精确恢复。

### 3.4 并发与一致性保障

- `decreaseCredits` 与 `increaseCredits` 均包裹在数据库事务（或 Supabase RPC）内执行，流程如下：
  1. 使用聚合结果只挑选本次扣费需要的池集合，并对这些池对应的原始 `credits` 行执行 `SELECT ... FOR UPDATE`，避免锁定整个用户的积分池。
  2. 通过统一的事务助手在同一事务中写入扣费或退款流水，确保扣减与持久化原子化。
  3. 事务提交后再向上游返回 `DeductResult`，避免读取到未提交的状态。
- 失败重试需保证幂等：使用 `trans_no` 或业务请求 ID 约束重复落库，并在检测到锁冲突时做指数退避重试。

## 4. 性能与可扩展性评估

- 数据库端聚合：所有余额聚合与筛选都在 Supabase/Postgres 层完成，事务内只会锁定被挑选出来的积分池，不再把整批流水回传应用层，大幅降低 CPU、内存与网络开销。
- 复杂度：聚合查询在 `WHERE user_uuid = $1` 下顺序扫描该用户的有效流水，复杂度为 `O(n_user)`；应用层只需处理可用池子集合 `O(k)`，在事务内写入扣费流水即可。
- 索引优化：
  - 新建 `idx_credits_pool ON credits(user_uuid, order_no, expired_at, credits)`，覆盖聚合与锁定所需列，确保扫描路径稳定且无需回表；必要时按 `expired_at` 建辅索引优化 FIFO 排序。
  - 上线前运行 `EXPLAIN ANALYZE` 验证慢查询，确保新索引命中；定期 `VACUUM/ANALYZE` 保持统计信息新鲜。
- 监控：增加慢查询告警，持续评估大户用户（万级扣费流水）下的执行时间，并在必要时把聚合逻辑封装为数据库函数/视图以进一步优化。

## 5. 实施步骤

1. **服务层改造**：重构 `services/credit.ts` 的扣费与聚合逻辑，补充类型定义与错误处理。
2. **API 调整**：
   - `app/api/image-generation/submit/route.ts`
   - `app/api/ai-callback/[provider]/route.ts`
   - 其他使用扣费能力的接口逐一核对，确保 metadata 持久化。
3. **事务封装**：在数据访问层实现统一的事务辅助方法，所有扣费/退款入口迁移到新方法。
4. **测试用例**：新增/更新单元测试与集成测试，覆盖单池、多池、跨池扣费与退款失败回滚场景。
5. **监控上线**：发布后开启 SQL 监控与异常报警，观察 24-48 小时，确认无新增负债池记录。

## 6. 测试策略与用例

### 6.1 单元测试（services 层）

1. **单池扣费**：
   - 准备一个余额为 100 的积分池，扣减 30，校验返回 `pools[0]` 信息正确，数据库余额变成 70。
2. **跨池扣费（顺序扣减）**：
   - 用户有两个池：A(余额 40, 先到期)、B(余额 80)。扣减 60 后应消耗 A=40、B=20。
3. **跨池扣费（不足余额报错）**：
   - 总余额 50，尝试扣减 60，应抛出 `not enough credits` 并保持数据库不变。
4. **过滤负债池**：
   - 数据库里存在 A(-10)、B(50)，扣减 20 时只允许从 B 池扣减。
5. **过期积分过滤**：
   - 构造一条已过期流水，确认聚合结果不包含该池。
6. **返回值幂等性**：
   - 同一次请求重复调用（模拟重试），使用相同 `trans_no` 保证只写一次扣费流水，或确保重复调用会检测到幂等约束。

### 6.2 集成测试（API 层）

1. **图片生成扣费 + 回调退款（成功路径）**：
   - 模拟 `/submit` 扣费成功并写入 metadata，再触发回调验证退款回原订单；检查 `credits` 表退款流水携带原 `order_no/expired_at`，且无新池被创建。
2. **图片生成扣费流程失败回滚**：
   - 在扣费后、事务提交前故意让后续步骤抛错，确认事务整体回滚，既无扣费流水也不污染 metadata。
3. **视频生成扣费 + 回调退款**：
   - 复用视频生成接口完整走通扣费、metadata 持久化与回调退款，确保与图片流程一致并复用扣费返回值。
4. **并发扣费竞争**：
   - 模拟同一用户多次并发扣费（例如 5 并发），验证最终扣减总额正确、无负债池产生，并观察锁冲突是否按退避策略重试成功。
5. **退款缺失 metadata 的错误提示**：
   - 构造 metadata 缺失场景，确认退款接口返回明确错误并拒绝创建新积分池，日志具备排查信息。
6. **API 幂等重试**：
   - 使用相同业务请求 ID（`trans_no`）重复提交扣费与退款请求，确保幂等保护避免重复扣减或退款。

### 6.3 数据库层测试

1. **聚合函数/视图验证**：
   - 直接调用 `groupCreditsByPool` 所使用的 SQL（或视图），对比返回结果与手动聚合的余额一致。
2. **索引效果**：
   - 在测试环境跑 `EXPLAIN ANALYZE`，确认新索引被命中，查询不再触发 Seq Scan。
3. **锁粒度验证**：
   - 在数据库层发起两个事务，各自锁定不同池，确认不会互相阻塞；若锁同一池则阻塞行为符合预期。

### 6.4 观察与监控

- 上线灰度期间开启慢查询日志与错误告警，重点关注：
  - 是否仍出现 `order_no` 负余额扣费场景。
  - 是否出现锁超时或重试次数激增。
  - Metadata 缺失导致退款失败的请求占比。

---

## 6. 验收与回归

- 扣费后立即查询用户积分流水，确认不存在余额为负的池。
- 模拟失败流程触发退款，检查新流水是否绑定原 `order_no` 且未新增独立池。
- 并发压测：模拟多请求同时扣费同一用户，确保最终余额符合预期，且无重复扣费日志。
- 日志审计：扣费失败/退款失败需输出统一格式的结构化日志，便于排查。

## 7. 风险与应对

- **事务冲突**：同用户高并发扣费可能导致事务重试，需要在应用层实现有限次数的退避重试机制。
- **调用方未适配**：若某个业务线未持久化返回的池信息，会在退款时收到错误，需要在上线前完成接口巡检并列出白名单。
- **历史负债影响展示**：虽然本方案不修复旧数据，但前端展示若直接聚合所有流水，仍会看到负余额池；需要在展示层按新逻辑聚合或标注历史数据待处理。

---

> 本方案聚焦“阻止新问题产生”，后续若需处理历史数据，可在此基础上另行制定迁移策略。
